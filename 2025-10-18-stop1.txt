⏺ Key Findings from MCP Study

  What MCP Actually Is

  MCP is a standardized protocol for exposing tools to LLMs via a client-server architecture:

  ┌─────────────────┐
  │  LLM API Call   │ (Claude Messages API)
  └────────┬────────┘
           │
  ┌────────▼──────────────┐
  │  Integration Layer    │ (Claude Agent SDK)
  │  - Manages MCP servers│
  │  - Discovers tools    │
  │  - Translates schemas │
  │  - Proxies execution  │
  └────────┬──────────────┘
           │
  ┌────────▼────────┐
  │  MCP Servers    │ (Slack, GitHub, filesystem, etc.)
  │  - Expose tools │
  │  - Run as       │
  │    subprocesses │
  └─────────────────┘

  Critical Insight: Providers DON'T Need MCP Support

  The LLM provider API (Claude's Messages API) doesn't know about MCP.

  From the Claude Agent SDK documentation:
  - The SDK acts as the integration layer between MCP servers and Claude's API
  - It translates MCP tool schemas to Claude's tool calling format
  - Claude's Messages API receives standard tool definitions, not MCP-specific ones
  - The SDK manages MCP server lifecycle and proxies tool execution

  You Were Right to Push Back on "Bridge"

  "Bridge" sounds like a simple proxy, but MCP integration is more substantial:

  1. Server Lifecycle Management: Start/stop MCP server subprocesses
  2. Tool Discovery: Connect to servers, list available tools via JSONRPC
  3. Schema Translation: Convert MCP tool schemas to LLM-specific format
  4. Execution Proxy: When LLM calls tool, route to correct MCP server
  5. Result Handling: Translate MCP responses back to LLM format

  Where MCP Should Live in Amplifier

  Looking at Amplifier's architecture and the Claude SDK model:

  The Providers DON'T Need to Change

  Your Anthropic provider already does:
  def _convert_tools(self, tools: list[Any]) -> list[dict[str, Any]]:
      for tool in tools:
          anthropic_tools.append({
              "name": tool.name,
              "description": tool.description,
              "input_schema": input_schema,
          })

  It already handles arbitrary tools - it just reads their schemas and translates to Anthropic's format. MCP
  tools would work the same way.

  MCP Support Should Be a Tool Module

  Create amplifier-module-tool-mcp (not "bridge") that:

  1. Manages MCP servers as resources (like how filesystem tool manages files)
  2. Wraps each MCP tool as an Amplifier Tool
  3. Handles JSONRPC communication with servers
  4. Provides tool discovery - dynamically register tools from configured servers

  Why separate from providers?
  - MCP is provider-agnostic (works with any LLM that supports tool calling)
  - Keeps providers clean (they're LLM API adapters, not tool managers)
  - Follows Amplifier's separation of concerns

  The Integration Pattern

  # In amplifier-module-tool-mcp
  class MCPToolWrapper:
      """Wraps an MCP server tool as an Amplifier Tool."""

      def __init__(self, mcp_client, server_name, tool_def):
          self.name = f"mcp_{server_name}_{tool_def.name}"
          self.description = tool_def.description
          self.input_schema = tool_def.inputSchema  # From MCP server
          self.mcp_client = mcp_client
          self.server_name = server_name
          self.tool_name = tool_def.name

      async def execute(self, input: dict) -> ToolResult:
          # Proxy to MCP server via JSONRPC
          result = await self.mcp_client.call_tool(
              self.server_name,
              self.tool_name,
              input
          )
          return ToolResult(success=True, output=result)

  Then:
  1. Orchestrator passes all tools (including MCP tools) to Provider
  2. Provider translates schemas to LLM format (doesn't care they're from MCP)
  3. LLM requests tool execution
  4. Orchestrator calls tool.execute() (which proxies to MCP server)

  So: Do Providers Need Changes?

  NO. Providers already handle arbitrary tools. They don't need to know about MCP at all.

  The tool module handles MCP complexity, and providers just see regular Amplifier Tools with standard schemas.

  Is this making more sense now? The key insight is that MCP integration is about tool provision, not provider
  modification.

  6. Resolution Priority (Matching Amplifier's Pattern)

  When loading MCP servers:

  1. Profile inline servers (highest priority) - immediate context
  2. Project .amplifier/mcp.json - team/project specific
  3. User ~/.amplifier/mcp.json - personal servers
  4. Environment $AMPLIFIER_MCP_CONFIG - override for testing

  Servers with same name in higher priority locations override lower ones.
